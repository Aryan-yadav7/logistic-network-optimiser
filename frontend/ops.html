Pass 1 — Quick Sort by Priority
Partitions the entire order queue around a pivot priority value. All priority-1 orders are placed before priority-2, priority-2 before priority-3, and so on. Quick Sort is used here because we need a fast in-place partition and stability is not required — same-priority ties will be resolved in Pass 2.
O(n log n) avg · O(n²) worst · Unstable
Pass 2 — Merge Sort by Deadline (per group)
Merge Sort is applied independently to each priority bucket produced by Pass 1. Within each bucket, orders are sorted by deadline ascending (earliest deadline first). Merge Sort is chosen because it is stable — packages already in priority order remain so, and only the within-group deadline ordering changes. Result: globally sorted by (priority ASC, deadline ASC).
O(n log n) worst · Stable · Extra O(n) space
